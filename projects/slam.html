Semantic SLAM Acceleration

A hardware acceleration framework for real-time semantic SLAM, targeting the AI/ML kernel on an AMD Versal adaptive SoC. By offloading neural network inference and optimizing system-level integration, we achieved a 14.5x speedup in inference and a 120.6x improvement in energy efficiency, easily meeting the strict 10 Hz LiDAR real-time constraint.

Architecture at a Glance:

  • Inference Acceleration: Mapped RangeNet DarkNet53 to the AIE-ML architecture using INT8 quantization and fused operators.
  • System Integration: Bypassed ROS2 message passing with a custom, zero-copy shared-memory ring buffer (IPC).
  • CPU Optimization: Parallelized SG-SLAM graph construction using Intel TBB multithreading.
  • Performance: Reduced end-to-end latency from 292.4 ms to 42.3 ms, and power from 222.0 W to 12.7 W.

The Challenge: Real-Time SLAM on the Edge

Modern semantic SLAM pipelines rely on deep neural networks (like RangeNet++) to identify dynamic objects and map unstructured environments. However, a pure PyTorch GPU implementation takes ~231.4 ms just for inference, completely failing the 100.0 ms constraint of a standard 10 Hz LiDAR sensor.

To deploy this on edge-constrained autonomous systems, we had to systematically eliminate bottlenecks across the entire pipeline—from the silicon level up to the operating system.

Methodology & Implementation

1. AIE-ML Kernel Acceleration (RangeNet++)

We bypassed a traditional FPGA-fabric systolic array (which lacked the DSP budget for this workload) and targeted the Versal AIE-ML array. The design maps convolutions to tiled INT8 matrix multiplications using hardware-native intrinsics.

  • INT8 Quantization & Fusion: Batch normalization was folded directly into convolution weights. We utilized per-channel INT8 activation scaling to enable 8-bit multipliers, achieving a 4x model-size reduction (201 MB to 50 MB) with only a ~1.1% accuracy drop.
  • Kernel Design: Implemented 4 core kernels (3x3 conv, stride-2 3x3 conv, 1x1 conv, and element-wise addition).
  • Execution: The RangeNet input was divided into spatial tiles of 8x32 pixels, processed as 32-channel blocks through the AIE-ML matrix multiply unit.

2. Shared-Memory IPC (Bypassing ROS2)

Upstream SG-SLAM relies on offline, file-based input. To make it a live pipeline, we built a custom rangenet_sgslam_ipc bridge to completely bypass ROS message serialization.

  • Zero-Copy Ring Buffer: Sends each labeled LiDAR scan as one complete frame through a POSIX shared-memory ring (8 slots, 200k points/slot).
  • Drop-Oldest Policy: Keeps the producer nonblocking at the LiDAR rate. If the consumer falls behind, the producer overwrites the oldest, non-held frame to prioritize fresh data.
  • Latency Impact: Eliminated the ~20.0 ms per-frame file transfer overhead present in the baseline software stack.

3. Multithreaded Graph Construction

With inference accelerated, graph construction became a major CPU bottleneck. The $O(N^2)$ all-pairs distance pass in BuildGraph() was checking every node pair against edge thresholds serially.

We parallelized this using Intel TBB. The main loop was converted to a tbb::parallel_for, where each worker threads its own row of the adjacency, edge, and embedding matrices to avoid shared write conflicts. This improved graph-construction throughput for 512 nodes from 6.80 FPS to 10.64 FPS.

4. Attempted ICP Registration Offload

We attempted to port the Iterative Closest Point (ICP) correspondence search and normal-equation accumulation ($J^T J$ and $J^T r$) to a hardware kernel. However, the outer ICP loop remained heavily sequential and dependency-bound to the host CPU. Benchmarking proved this math was not the dominant bottleneck, so the software backend was retained to preserve algorithmic fidelity.

Results & Performance Analysis

Benchmarking was conducted using sequence 00 from the KITTI dataset. Baseline metrics were gathered via an AWS g5.xlarge instance (NVIDIA A10G), while accelerated projections utilized AMD Power Design Manager (PDM) and cycle-approximate AIE simulation.

Full-Network Latency

The accelerated run cuts total pipeline latency by 6.9x, providing ~57.7 ms of headroom under the strict 100.0 ms frame budget.

Energy Efficiency & Power

By mapping the heavy lifting to the AIE-ML array, power consumption dropped dramatically compared to the discrete GPU baseline, yielding a massive 120.6x improvement in overall energy-per-frame efficiency.
